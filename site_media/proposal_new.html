Custom Web Form Generation Proposal

UPDATE ON THE BACKEND IMPLEMENTATION:
In my proposal I suggested using a a single table for actual data of the form. But that might go ineffcient in cases when at times there are a large no of forms active and Hence, none of them can actually be sent to the NOSQL database for storing. For the same I suggest the following implementation.

In the first table i would like to keep it same and store the private and public JSON string as this is very fast regarding the serving of the form for filling purpose.
I would now not have a second table as suggested before. Instead in the form row in table1 I will have a column for storing the name of the table in which the form data for that concerned form is stored. That means i would have different copies of the table2. The table name would be generated dynamically on the basis, if the last table has lesser no of rows filled at that time. If the last table has less no of rows, We can put the the data in same table else create a new table. and accordingly set it in table1 for that form so that it is easy to refer which table needs to be opened for serving the concerned form.

While creating the form, The creator is asked for the no of form submissions he expects.

Serving the form is fast due to the JSON strings in table1. This also keeps its flexibility intact to add new features as discussed in original proposal. The Private string also keeps the feature to provide powerful data validity intact. The additional feature we get is that as the forms are clustered in groups we get lesser no of Tables created. As the user tells the no of form submissions he expects, This would be useful in deciding in whether a new table should be created for the form or stored in the previous table. As the user gives an expectation of the number of form submissions, this would also help in preventing any form spamming as the creator may be warned that the no of submissions is going much above the expected number and the form submission can be temporarily held down ( This is other than showing the captcha when there is a huge traffic from a given IP ).

MongoDB supports does not support compression but HBase (another NOSQL database) supports compression. Hence HBase might be preferred to save space.

1. Storing attributes in a JSON object: For the various attribute of the document element, I would store them in a JSON object. These JSON object will stores the required properties which will be served as such for rendering the form for submission. These objects are stored after using the stringify operation. As we have easy features to parse them in javascript (for front-end) and also in django(for backend). In the demo, I have already used this and it works good. We can store the same into two parts. One containing the form public data which would be used for rendering the form and the other containing the private data which would be present only in backend and used for validation checking etc.

Implementation: While changing any element store their property in the json object too. Then stringify() the object and store in the database. Also JSON.parse() can be used for converting string to JSON object. At backend we can import the simplejson library and use the loads() and dumps() method to achieve the same functionality.

Backend Implementation:

17. For backend we can have two tables:

Table 1 has the following fomat:

    || Id || Title || Public JSON string || Private JSON string || No of submissions || start time of submission || last time of submission ||  Id of creator ||

Table 2 has following fomat:

   || Id of user who submitted the form || Foreign key to Table 1 || Unique identity of field || Value of the field ||

The JSON Public String is mainly used for rendering the form for submission at the user end. The Private JSON string is used for validation of the form and other analysis purposes as discussed before. Both the fields are generated at time of Form creation. The form can be filled only after the start time and ends after the last time of submission. Another way is to create an intermediate table which holds the field and each form is linked to the fields it has. Then each field is in turn linked to its values in the third table. But this model would increase the time in serving the form as in first case we just need to send the public JSON string and the form is created. In other case we have to get the JSON string of each field and serving the form will take time. Also while analysis of the form, the second will need to perse each fields JSON string and then proceed however in the first model we have a single string. But as the string size have a huge difference, any model would work fine. So its better to have two tables than having three tables and increasing the linking.

Advantages of the model:

1. Fexibility: The proposed model is highly flexible in terms of the adding new features to existing form elements or adding a new type of form element. As this would not require editing anypart of the database. We just need to store it together with the json string and then parse it as we need. The new variable will be present in it. Hence, Any changes in user GUI will not affect the backend. We just need to edit the page which displays the form to include the new feature too.

2. Backward Compatibility: If we add some features, then as we are not adding a new table column etc, we can check easily in the parsed string which data is present and not. In older versions, the database's json data will have no field in that part and hence we can either switch to default for that field or ignore it as required.

3. Listing the forms: As some major parameters of the form ; the Form Title, Form Id, No of submission, date of creation and submission and details of the creator etc are not encoded in the JSON string, hence in cases where we just need to present a summary of the form, eg while listing the form, we dont need to parse the whole form's JSON string. Hence as these field may be readily required they are stored without encoding in the JSON string so that we do not require to parse the whole string just for displaying the summary of the form.

3. Query Response Time: As the no of rows go on increasing with the no of submissions of the form, the queries may take time after the no of rows increase to a very large number. For this, we can have a scheduled time after which a form submission is stopped and also a time after which the form is completely deleted from the server(though some forms may not have any such time). After a form submission is stopped the form needs no more editing. Hence, we can have them stored in some NoSQL database like mongodb (thanks to adam and dhruv for the idea). MongoDB speeds up the process to a lot extent. Also integration of mongodb with django is very easy through the opensource django project on:
https://github.com/hmarr/django-mumblr
