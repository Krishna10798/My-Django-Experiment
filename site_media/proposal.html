1. Positioning elements with drag feature. The label of any element can be dragged to start positioning the element. This will enable the user to position elements in the form anywhere on the workspace.(Sample present in demo.) To be improved so that the element cannot be dragged outside the user viewable frame.

Implementation: While setting the drag 'handle' in 'draggable()', properly set the 'containment' attribute (in most cases to 'parent').

2. Reordering elements in which we have constraints on movement of the objects. For example, after you create a radio box and add some radio buttons, You can reorder the radios by dragging them. However the radios can't be moved anywhere on the form. They can only be moved in group by dragging their main heading label.(Present in the demo)

Improvements: Allow the user if the radio buttons should be arranged horizontally or vertically. The User can use the rearranging property without allowing to seperate them.

Implementation: The dispaly for horizontal alignment should be removed from inline to block type. Then dragging should be allowed in constrained region by properly setting the 'containment' attribute.

3. Asynchronous loading of elements: There are UI elements which may slow down the loading of the page. Hence, these features can be loaded asynchronouly after the form has loaded most of the required content to present user with a basic editing layout. We can also try loading some features only if the user wants to load them.

4. Display context menu to edit elements on a right click. The user should get a shortcut menu upon click where he can delete elements and can edit other features too.

Implementation: There are many easy approaches to it but the best and fastest i could find was the context menu plugin to jQuery. As we have used jQuery for whole of our project, its better to use this than switching to some other UI since we have to care about performance and speed. This being a plugin to our existing imported library is light to serve our purpose.

The plugin page is: http://www.javascripttoolbox.com/lib/contextmenu/

The interface provided by them for adding these functionalities is very good and easily implementable. We can load this feature asynchronously.

5. Addition of a color picker widget so that user can choose colors for different elements of the form such as background color, Title color etc. This can also be used as an input element for people.

Implementation: Again as we opt for a simple color picker plugin for jQuery. This is light yet powerful and fully functional for our use.

This is the plugin i use for this purpose. http://www.eyecon.ro/colorpicker/

7. Dependent Elements: I would also like to add support for dependent elements. For example, There are two combo boxes. The valid contents in the second one depends on the content in the first one. This will ofcourse be added by handling the onchange event of the combo box. Similarly there are forms which are divided into several parts. If time permits then we can have this feature too by making the form tabbed and allowing progressing only if one part is completly filled and checked for validity.

8. Custom Templates: These are predefined css templates which can be used to set the UI colors and layouts in a single go. This will not only save space as each form will just link to these template instead of seperately defining them for each form but also provide a big ease to the user(form creator).The Framework can be easily made in such a manner that more templates can be extended without affecting the precious forms.

9. Resizing: The form elements like the size of the entire form etc should be resizable by providing a drag handle on the boundary and at the corners. Same should be the case with other possible elements where we may feel the necessity. As an example, see the demo where textarea and form size is resizable.

10. Tool Tip: There are situations when form creators need to give some hint of what should be the input in a form field. This would enable the user to add tooltips as easily as possible.

The number field will have statistical analysis on them like mean, statndard deviation or can also have different graphs plotted for output.

Implementation: Number fields can be checked for validity on derver as well as client side. The slider is already present in the Jquery and can be easily integrated.

12. Validity Of Input: We can validate the input on form submission at client as well as server side. Some validations are already added to the form on basis of type of input. Also checks like the input to a combo box or radio button is amongst its option as we have to necessarily deal with the DOM manipulations. Hence the type of field will decide the type of check to be done.

Backend Implementation:

16. For backend we can have two tables:

Table 1 has the following fomat:

      || Id || Title || Json String of the fields || No of submissions || Date of creation || last date of submission ||  Id of creator ||

Table 2 has following fomat:

      || Id of user who submitted the form || Foreign key to Table 1 || Unique identity of field || Value of the field ||

Advantages of the model:

1. Fexibility: The proposed model is highly flexible in terms of the adding new features to existing form elements or adding a new type of form element. As this would not require editing anypart of the database. We just need to store it together with the json string and then parse it as we need. The new variable will be present in it.

2. Backward Compatibility: If we add some features, then as we are not adding a new table column etc, we can check easily in the parsed string which data is present and not. In older versions, the database's json data will have no field in that part and hence we can either switch to default for that field or ignore it as required.

3. Query Response Time: As the no of rows go on increasing with the no of submissions of the form, the queries may take time after the no of rows increase to a very large number. For this, we can have a scheduled time after which a form submission is stopped and also a time after which the form is completely deleted from the server(though some forms may not have any such time). After a form submission is stopped the form needs no more editing. Hence, we can have them stored in some NoSQL database like mongodb (thanks to adam and dhruv for the idea). MongoDB speeds up the process to a lot extent. Also integration of mongodb with django is very easy through the opensource django project on:
